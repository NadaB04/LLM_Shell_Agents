## What this project is

`doit` is a single-file LLM-powered shell agent CLI (university assignment). It has no build step, no test suite, and no package structure â€” the entire implementation lives in the file `doit` (no extension).

## Running

```bash
# Windows â€” run directly with python (most reliable)
python doit "your request"

# Windows PowerShell via wrapper
doit "your request"

# Git Bash / Linux / Mac (requires chmod +x doit once)
./doit "your request"
```

Flags: `--history`, `--memories`, `--forget KEY`, `--clear-history`, `--agentic "GOAL"`

For multi-turn testing across invocations, set a stable session ID:
```bash
DOIT_SESSION_ID=mysession python doit "your request"
```

## Config

`~/doit.cfg` (JSON) â€” must exist before running. Three configurations are used in this project:

```json
// API model (fast, cloud)
{ "model": "groq/llama-3.1-8b-instant", "api_key": "gsk_...", "api_base": "", "model_type": "api" }

// Local with tool-calling support (ollama must be running)
{ "model": "ollama/qwen3:4b", "api_key": "", "api_base": "http://localhost:11434", "model_type": "local_tools" }

// Local instruction model, no tool-calling
{ "model": "ollama/gemma3:4b", "api_key": "", "api_base": "http://localhost:11434", "model_type": "local_no_tools" }
```

Local models are downloaded: `ollama list` shows `qwen3:4b` (2.5 GB) and `gemma3:4b` (3.3 GB).

## State files

- `~/.doit/history.json` â€” conversation history, keyed by `session_id`
- `~/.doit/memory.json`  â€” persistent key/value user memories

## Architecture

All logic is in `doit`. The flow per invocation:

1. `load_config()` â†’ reads `~/doit.cfg`
2. `load_history()` + `load_memory()` â†’ reads `~/.doit/`
3. `read_shell_history()` â†’ reads `~/.bash_history` or `~/.zsh_history`
4. `_build_messages()` â†’ assembles system prompt + last 10 session turns + current request
5. `_call_llm()` â†’ LiteLLM call, parses JSON response
6. `_process_response()` â†’ dispatches on `resp["type"]`
7. Appends history entry, `save_history()`

### LLM response types

The LLM always returns one of these JSON shapes (defined in `_SYSTEM_PROMPT`):

| type | handler | notes |
|---|---|---|
| `command` | `handle_command()` | dangerous=true â†’ y/N prompt before running |
| `answer` | print text | for how-to questions, jokes, etc. |
| `impossible` | print reason | request can't be done as a shell command |
| `clarification` | `handle_clarification()` | asks user, then re-calls LLM with answer |
| `multi_step` | `handle_multi_step()` | sequential commands, stops on first failure |

Any response may include `"memories": [{"key": "...", "value": "..."}]` to update `memory.json`. Empty value = forget.

### Agentic loop (`--agentic`)

Uses a separate system prompt (`_AGENTIC_SYSTEM`) and a different response schema (`agentic_step` / `agentic_done` / `agentic_ask`). The loop runs in `run_agentic()`, appending execution output back into `messages` after each step. Max 10 steps.

### Session isolation

`DOIT_SESSION_ID` env var (default `"default"`) scopes history. Set in `~/.bashrc`:
```bash
export DOIT_SESSION_ID=$(uuidgen)
```
History is filtered to the current session when building messages; all sessions share the same `history.json` file so cross-session references are possible.

## Windows-specific issues

- **Shell execution**: `run_shell()` detects Git Bash at `C:\Program Files\Git\bin\bash.exe`. Commands run through bash but Windows paths sometimes cause `exit 126` â€” this is a Windows/bash interop issue, not an LLM error.
- **Unicode**: All non-ASCII characters in `print()` calls (e.g. `âš `, `â”€â”€`) have been replaced with ASCII equivalents (`[!]`, `--`) to avoid `UnicodeEncodeError` on `cp1255` terminals. Do not re-introduce Unicode in print statements.
- **Memory trigger phrasing**: The Groq llama-3.1-8b model does not reliably use the `memories` JSON field from natural requests like "remember my name". Use explicit phrasing: `"store in your memories that key is value"`.
