Assignment 3: Agents — Design Questions & Answers
====================================================
(All answers refer to the implementation in `doit`)


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SECTION 1 — Single command at a time
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How do you capture the model's command output in a convenient format?

A: The LLM is instructed (via the system prompt) to always respond with a
   single JSON object. The `type` field tells the program what kind of
   response it is. For an executable command:

       {"type": "command", "command": "ls ~/Documents", "description": "...", "dangerous": false}

   This avoids any ambiguity between "a command" and "an explanation" — the
   program just reads `resp["type"]` and dispatches accordingly. The JSON is
   parsed in `_call_llm()`:

       content = response.choices[0].message.content.strip()
       # strip optional markdown fences
       content = re.sub(r"^```(?:json)?\s*", "", content)
       content = re.sub(r"\s*```$", "", content)
       return json.loads(content)

   If JSON parsing fails, a regex `\{.*\}` fallback extracts the first
   JSON-like block, and if that also fails the raw text is wrapped as
   `{"type": "answer", "text": content}`.


Q: What happens if the user requests something that is not possible to do
   in the shell?

A: The LLM responds with `type: impossible`:

       {"type": "impossible", "text": "<reason>"}

   The system prompt explicitly lists this case:

       4. Cannot fulfill:
       {"type":"impossible","text":"<reason>"}

   The handler in `_process_response()` prints the reason:

       elif rtype == "impossible":
           print(f"\n[Cannot do that: {resp.get('text', '')}]")


Q: What happens if the user asks something like "tell me a joke" or
   "what can you do"?

A: Non-command questions produce `type: answer`:

       {"type": "answer", "text": "<explanation or answer>"}

   The system prompt says:
       3. Informational answer (no execution needed):
       {"type":"answer","text":"<your explanation>"}

   This also covers how-to questions ("how do I undo the last git commit?")
   — the LLM recognises them as informational and answers in plain text
   rather than generating a command to execute.

SECTION 2 — Identifying dangerous commands
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How do you identify dangerous commands?

A: The classification is done by the LLM as part of the same response — no
   separate call is needed. The system prompt defines exactly which command
   patterns are dangerous vs. read-only:

       Dangerous = true for: rm, rmdir, mv, cp (overwrite), chmod, chown,
       kill, pkill, sudo, package managers (apt/pip/brew), git push/reset
       --hard, curl|sh, dd, mkfs, truncate, write redirects (> file),
       systemctl stop/disable, and anything else that alters state.

       Dangerous = false for: ls, cat, head, tail, grep, find, ps, df,
       du, pwd, echo, which, man, stat, file, wc, diff, git log/status/diff,
       and other read-only operations.

   When `dangerous` is true, `handle_command()` pauses before executing:

       if dangerous:
           print("\n⚠  This command modifies the system.")
           ans = input("  Execute? [y/N] ").strip().lower()
           if ans != "y":
               print("Skipped.")
               return None

   Combining detection and classification into one LLM call (instead of
   two) keeps latency low and avoids the case where the two calls disagree.

SECTION 3 — Model flexibility
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How is switching between a hosted API model, a local model without tool
   calling, and a local model with tool calling supported?

A: LiteLLM is used as a uniform API layer. All three model types go through
   the same `_call_llm()` function:

       def _call_llm(config: dict, messages: list) -> dict:
           kwargs = {
               "model":       config["model"],
               "messages":    messages,
               "temperature": 0.1,
           }
           if config.get("api_key"):  kwargs["api_key"]  = config["api_key"]
           if config.get("api_base"): kwargs["api_base"] = config["api_base"]
           response = completion(**kwargs)
           ...

   The `~/doit.cfg` file selects the model and type:

       {
         "model":      "ollama/qwen3:4b-instruct",
         "api_base":   "http://localhost:11434",
         "model_type": "local_no_tools"
       }

   Because all three model types are asked to produce JSON-in-prompt (not
   native function calls), even a model without tool-calling support works
   correctly. The `model_type` field is available for future use (e.g., to
   switch to native function-calling for `local_tools` models), but the
   JSON-in-prompt approach is the universal baseline that works for all
   three.

   For Ollama: set `model` to `"ollama/modelname"` and `api_base` to
   `"http://localhost:11434"`. LiteLLM handles the translation.

SECTION 4 — Multi-turn
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How and where do you store the history?

A: History is stored in `~/.doit/history.json` as a JSON array. Each entry
   is a dict:

       {
         "id":        "<uuid>",
         "session_id": "default",        # from $DOIT_SESSION_ID
         "timestamp": "2025-06-10T...",
         "cwd":       "/home/user/proj",
         "request":   "list the files",
         "response":  {"type": "command", "command": "ls", ...},
         "execution": {"stdout": "...", "stderr": "", "returncode": 0}
       }

   After each invocation, the new entry is appended and the file is
   rewritten with `save_history()`.


Q: How do you present the history to the LLM?

A: In `_build_messages()`, the last 10 entries for the current session are
   converted to a proper `messages` array with alternating user/assistant
   turns. Crucially, execution output is appended as a `[Execution result]`
   user message so the LLM knows exactly what stdout/stderr/returncode
   resulted from each command:

       for entry in session_entries[-10:]:
           messages.append({"role": "user",      "content": entry["request"]})
           messages.append({"role": "assistant",  "content": json.dumps(entry["response"])})
           ex = entry.get("execution")
           if ex:
               feedback = f"[Execution result — returncode: {ex['returncode']}]"
               if ex.get("stdout"): feedback += f"\nstdout:\n{ex['stdout'].rstrip()}"
               if ex.get("stderr"): feedback += f"\nstderr:\n{ex['stderr'].rstrip()}"
               messages.append({"role": "user", "content": feedback})


Q: How do you distinguish between new commands and commands that refer to
   previous ones?

A: This is handled entirely by the LLM. Because the full conversation
   history is included in the messages array, a follow-up like "now sort
   them by date" or "no, I meant latest first" is interpreted in the
   context of the previous turn — the same way a ChatGPT conversation
   works. The LLM sees the original `ls ~/Documents` command, its output,
   and then the follow-up, so it can produce a correct follow-up command
   without the user needing to repeat context.

SECTION 5 — Clarifications
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How do you identify when clarification is needed?

A: The LLM decides. The system prompt instructs it to use `type:
   clarification` when the request is ambiguous:

       5. Need clarification before acting:
       {"type":"clarification","question":"<your question>","options":["opt1","opt2"]}

   The LLM is also told:
       When a user says "execute it", "run it", "do it", or similar,
       execute the last suggested command.
       When a user asks a question like "how do I…", respond with
       type=answer, not type=command.

   This nudges it to only ask when genuinely unclear, not for every request.


Q: What do you do if the user does not answer the question?

A: `handle_clarification()` reads the user's answer with `input()`. If the
   user presses Enter with no text, the function returns early with no
   action. If the user presses Ctrl-C or Ctrl-D, `EOFError`/
   `KeyboardInterrupt` is caught and the function prints "Aborted." and
   returns:

       try:
           answer = input("\nYour answer: ").strip()
       except (EOFError, KeyboardInterrupt):
           print("\nAborted.")
           return None
       if not answer:
           return None


Q: How is the clarification mechanism implemented?

A: It is one of the JSON `type` values. No separate tool-calling mechanism
   is needed. When the dispatcher sees `type: clarification`, it calls
   `handle_clarification()`, which prints the question and options, reads
   the user's answer, and then calls `_call_llm()` again with the answer
   appended as a new user message. The response to that call is then
   processed normally (it could be a command, an answer, or even another
   clarification):

       def handle_clarification(resp, config, history, memory, shell_hist, session_id):
           ...
           answer = input("\nYour answer: ").strip()
           messages = _build_messages(history, memory, shell_hist, answer, session_id)
           resp2 = _call_llm(config, messages)
           return _process_response(resp2, ...)

SECTION 6 — Richer interactions
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How do you make it work for non-command questions and follow-ups like
   "execute it"?

A: The `type: answer` response type handles non-command questions. The LLM
   produces an answer in text form and it is printed directly. For follow-
   ups like "execute it" or "run it", the conversation history already
   contains the previous assistant response (including the `command` field),
   so the LLM re-reads it and produces `type: command` with the same
   command. The system prompt explicitly mentions this:

       When a user says "execute it", "run it", "do it", or similar,
       execute the last suggested command.


Q: What can happen in longer sequences of interaction?

A: The main risk is context drift — after many turns the LLM may lose track
   of which command is "the current one". This is mitigated by: (1) capping
   history at the last 10 session entries (`session_entries[-10:]`), so the
   most recent context is always present; (2) including execution output
   after each command so the LLM knows what happened. A future improvement
   would be context compaction (summarising old turns) to handle very long
   sessions gracefully.

SECTION 7 — Memory
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How are memories stored and when?

A: Memories are stored in `~/.doit/memory.json` as a flat key/value dict,
   e.g. `{"preferred_editor": "vim", "llm_class_folder": "~/school/llms/ass3"}`.
   The LLM can create, update, or delete a memory by including a `memories`
   array in any response:

       {"type": "command", ...,
        "memories": [{"key": "llm_class_folder", "value": "~/school/llms/ass3"}]}

   The `apply_memories()` function processes these after every LLM response:

       def apply_memories(new_memories, memory):
           for m in new_memories:
               key, value = m.get("key","").strip(), m.get("value","").strip()
               if value.lower() in ("", "null", "none", "forget", "delete"):
                   memory.pop(key, None)
               else:
                   memory[key] = value
           return memory

   The user can also inspect and delete memories manually:
       doit --memories
       doit --forget preferred_editor


Q: How do memories appear in the LLM context?

A: The system message is built fresh for every invocation in
   `_build_messages()`. The memory dict is injected as a section near the
   top:

       if memory:
           lines = "\n".join(f"  {k}: {v}" for k, v in memory.items())
           sys_parts.append(f"\nUser memory (persistent facts):\n{lines}")

   This appears before the conversation history, so the LLM treats memories
   as stable, session-independent background knowledge.


Q: What about commands that refer to previous memories like "I changed my
   mind about the sorting order, ask me each time"?

A: Because the memory is injected on every call, the LLM can read it and
   react to a natural-language instruction like "I changed my mind — delete
   that preference". The LLM would respond with `type: answer` or
   `type: command`, and include `"memories": [{"key": "sort_order",
   "value": ""}]` to delete the old value. The system prompt signals that
   an empty value means "forget".

SECTION 8 — User awareness
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How do you integrate the user's manual shell actions in the context?

A: `read_shell_history()` reads the appropriate history file
   (`~/.bash_history` or `~/.zsh_history`) and returns the last 15
   commands. zsh extended history lines (`: 1234567890:0;cmd`) are parsed
   to extract just the command part:

       if re.match(r"^: \d+:\d+;", line):
           line = line.split(";", 1)[1]

   The list is then injected into the system message:

       if shell_hist:
           lines = "\n".join(f"  $ {c}" for c in shell_hist)
           sys_parts.append(f"\nRecent manual shell commands (for context):\n{lines}")

   This lets the LLM answer questions like "summarise what I just did"
   (it sees `cd ~/school`, `mkdir data`, `python train.py` in the context)
   and also understand the current directory even if the user navigated
   there manually outside of doit.


Q: How do you distinguish between user's manual commands and doit's commands?

A: doit's own commands appear in the conversation history (the
   user/assistant turn pairs), while manual shell commands come from the
   shell history file. They are presented in separate sections of the system
   prompt, so the LLM can tell them apart:

   - Conversation history turns: "you (doit) previously ran `ls -la`"
   - Shell history section: "the user manually ran `cd ~/school`, `git pull`"

SECTION 9 — Output awareness
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How does the program answer questions about the previous command's output?

A: Each history entry stores the full execution result (stdout, stderr,
   returncode). When building messages for the next invocation, that result
   is appended as a user message right after the assistant's command
   response:

       ex = entry.get("execution")
       if ex:
           feedback = f"[Execution result — returncode: {ex['returncode']}]"
           if ex.get("stdout"): feedback += f"\nstdout:\n{ex['stdout'].rstrip()}"
           if ex.get("stderr"): feedback += f"\nstderr:\n{ex['stderr'].rstrip()}"
           messages.append({"role": "user", "content": feedback})

   So when the user asks "which of these looks safe to delete?" or "why did
   that command fail?", the LLM has the full output in its context window
   and can answer directly or generate a follow-up shell command.

SECTION 10 — Multi-tasking (multiple terminal windows)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How does the program behave sensibly when the user works in two terminal
   windows simultaneously?

A: Each terminal window gets its own session ID via the `DOIT_SESSION_ID`
   environment variable. To set it automatically when a terminal opens, add
   this to `~/.bashrc` or `~/.zshrc`:

       export DOIT_SESSION_ID=$(uuidgen)

   Or for a stable, window-specific ID:

       export DOIT_SESSION_ID="work"   # in one window
       export DOIT_SESSION_ID="docs"   # in the other

   Every history entry stores its `session_id`. The conversation history
   injected into each LLM call is filtered to the current session only
   (`session_entries = [e for e in history if e.get("session_id") == session_id]`),
   so "sort them by date" in Window 1 refers to Window 1's files, not
   Window 2's folders.

   For cross-session references like "do the same folder task we did in
   window 2 here", all sessions are stored in the same `history.json` file.
   A future improvement could pass the full history (with session labels) to
   the LLM when the user explicitly asks about "another window", so it can
   look across sessions.

SECTION 11 — Further extensions
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: Describe three extensions and implement one.

── Extension 1 (IMPLEMENTED): Autonomous agentic loop (--agentic flag) ──

Description:
   A single user request can trigger a fully autonomous, multi-step loop.
   The user gives a high-level goal; the agent decides which commands to
   run, executes them, observes the output, and decides what to do next —
   without the user having to issue each step manually.

Why it qualifies as a real agent capability:
   The LLM is in control of both planning and tool use. The user does not
   specify the steps — the agent figures them out. This is fundamentally
   different from a simple sequence of pre-planned commands.

How it is implemented:
   A separate system prompt (`_AGENTIC_SYSTEM`) defines three agentic
   response types:
     - `agentic_step`  → run a command, observe output, continue
     - `agentic_ask`   → ask the user a question, then continue
     - `agentic_done`  → goal reached, summarise and stop

   The loop in `run_agentic()` maintains a growing `messages` list. After
   each `agentic_step`, the execution result is appended and the LLM is
   called again:

       feedback = f"[Execution result — returncode: {result['returncode']}]"
       if result["stdout"]: feedback += f"\nstdout:\n{result['stdout'].rstrip()}"
       messages.append({"role": "user", "content": feedback})

   A max-steps guard (10 by default) prevents infinite loops:

       for step in range(1, max_steps + 1):
           ...
       else:
           print("[Reached maximum step limit]")

   Usage:
       doit --agentic "set up a Python venv and install requests and numpy"
       doit --agentic "find all TODO comments in this project and list the files"


── Extension 2 (described, not implemented): Context compaction ──

Description:
   For long-running sessions the history grows unboundedly. A compaction
   step would periodically (e.g., when history exceeds N entries) call the
   LLM to summarise older turns into a compact narrative ("The user moved
   to ~/proj, listed files, installed dependencies, and ran tests — all
   succeeded"), replacing them with a single summary entry. This preserves
   semantic context while keeping the prompt short.

Why it qualifies:
   It improves context management, which directly affects how well the agent
   handles long-horizon tasks. Without it, old context gets dropped
   (10-entry window) and the agent can "forget" important earlier actions.


── Extension 3 (described, not implemented): Project profiles ──

Description:
   Each directory could have a `.doit_profile` file (like CLAUDE.md /
   .cursorrules) containing project-specific instructions: preferred test
   command, deployment pipeline, style guide, known gotchas. When doit is
   invoked from a directory, it reads any `.doit_profile` found in the
   current dir or any parent dir and injects it into the system prompt as
   "Project context". This lets the agent behave differently in a Python
   project vs. a Rust project vs. a LaTeX writing project without the user
   having to repeat instructions every time.

Why it qualifies:
   It adds structured, persistent, directory-scoped memory — a real
   planning and context-management improvement that makes the agent more
   useful in practice.
